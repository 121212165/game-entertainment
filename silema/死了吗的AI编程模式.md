# 死了吗的AI编程模式

## 前言

通过复刻"死了么"APP的过程，我们发现这个项目之所以能够如此顺利地完成，不仅仅是技术选型的成功，更重要的是**AI使用模式的正确性**。本文总结在AI编程时代，如何高效地与AI协作完成开发任务。

---

## 🎯 AI使用的三层模式

### 第一层：交互模式（如何与AI对话）

### 模式1：规划-执行-验证循环

```
❌ 错误模式：直接让AI写代码
用户：帮我写一个签到APP
AI：[生成一堆代码]
用户：这个不对，改一下
AI：[修改代码]
→ 陷入循环修改，效率极低

✅ 正确模式：先规划再执行
用户：我要复刻"死了么"APP，功能包括...
→ AI拆解需求，列出任务清单
用户：确认规划
→ AI开始执行每个任务
用户：验证结果
→ 遇到问题，局部调整
```

**实际应用案例：**

在"死了么"项目中，我遵循以下流程：

1. **规划阶段**
   - 创建任务列表（11个步骤）
   - 明确每个步骤的输入输出
   - 确定技术栈：Kotlin + Jetpack Compose + Supabase

2. **执行阶段**
   - 逐个完成每个任务
   - 使用TodoWrite工具跟踪进度
   - 每完成一个就标记为completed

3. **验证阶段**
   - 编写SETUP_GUIDE.md供用户验证
   - 提供DEPLOYMENT_CHECKLIST.md检查清单
   - 确保每个模块都有明确的测试方法

### 模式2：渐进式信息揭示

```
❌ 一次性提供所有信息
用户：这是我的完整需求文档[10000字]
AI：[理解困难，容易遗漏关键点]

✅ 分阶段提供信息
阶段1：我要做什么（复刻"死了么"）
阶段2：技术栈选择（Kotlin原生）
阶段3：具体配置（已有Supabase账号）
→ AI在每个阶段都能聚焦关注
```

**实际应用案例：**

```kotlin
// 我没有一次性生成所有代码，而是分批生成：

批次1：项目结构
- 创建目录结构
- 生成配置文件（build.gradle.kts, settings.gradle.kts）
- 创建AndroidManifest.xml

批次2：数据层
- 数据模型（User.kt, CheckIn.kt）
- Supabase客户端（SupabaseClient.kt）
- 数据仓库（UserRepository.kt）

批次3：业务逻辑层
- SetupViewModel.kt
- CheckInViewModel.kt

批次4：UI层
- 主题配置（Color.kt, Theme.kt, Type.kt）
- 导航（Navigation.kt）
- 界面（SetupScreen.kt, CheckInScreen.kt）
```

这种分批生成的优势：
- 每个批次都聚焦在一个明确的主题上
- 批次之间有清晰的依赖关系
- 出错时容易定位和修复

### 模式3：上下文工程

```
核心原则：让AI看到"应该看到的"，隐藏"不需要看到的"

✅ 应该让AI看到的：
- 任务相关的文件
- 相关的技术文档
- 明确的需求说明

❌ 不应该让AI看到的：
- 不相关的项目文件
- 过时的代码
- 冗长的错误日志（除非相关）
```

**实际应用案例：**

```
✅ 正确的做法：
Read schema.sql (当前需要的文件)
Read rls_policies.sql (当前需要的文件)
→ 只读取与当前任务相关的文件

❌ 错误的做法：
Bash ls -R (列出整个项目目录)
Read 所有文件
→ 会引入大量无关信息，干扰AI判断
```

```
│  业务决策        │   代码生成            │

│  需求理解        │   格式转换            │
```

```yaml
  代码实现:
    - 生成所有Kotlin代码
    - 编写SQL脚本
    - 创建Edge Functions

  文档工作:
    - 编写README.md
    - 创建SETUP_GUIDE.md
    - 生成DEPLOYMENT_CHECKLIST.md

  配置文件:
    - build.gradle.kts
```

```
✅ 应该让AI做的：
1. 标准化代码编写
   - Data Class定义
   - Repository实现
   - UI组件代码
   - 配置文件模板

2. 文档和配置生成
   - README编写

   - API文档生成

❌ 不应该让AI做的：

   - 需要根据业务经验判断
   - 涉及长期维护成本
   - 影响团队协作

2. 敏感数据处理
   - API密钥管理
   - 安全策略制定
   - 加密算法选择

3. 性能优化
   - 需要实际测试数据
   - 需要考虑具体场景
   - 需要性能分析工具

4. 创意性工作
   - UI/UX设计
   - 产品定位
   - 营销策略
```

## 🚀 第三层：优化模式（如何提高效率）

### 优化1：并行执行模式

```
❌ 串行执行（慢）：
创建文件A → 等待 → 创建文件B → 等待 → 创建文件C
耗时：3分钟

✅ 并行执行（快）：
同时创建文件A、B、C
耗时：30秒
```

**实际应用案例：**

```xml
<!-- 一个消息中包含多个独立的Write调用 -->
<function_calls>
  <invoke name="Write">
    <parameter name="file_path">Color.kt</parameter>
  </invoke>
  <invoke name="Write">
    <parameter name="file_path">Theme.kt</parameter>
  </invoke>
  <invoke name="Write">
    <parameter name="file_path">Type.kt</parameter>
  </invoke>
  <invoke name="Bash">
    <parameter name="command">mkdir -p ui/screens</parameter>
  </invoke>
</function_calls>
```

**为什么能并行？**
1. 文件之间没有依赖关系
2. 每个文件的输入都是明确的
3. 不需要共享状态
4. 操作系统或AI可以同时处理

**并行执行的条件：**
```
✅ 可以并行：
- 创建独立的文件
- 执行独立的脚本
- 生成不相关的代码块

❌ 不能并行：
- 文件B依赖文件A的内容
- 需要先编译再测试
- 有明确的执行顺序要求
```

### 优化2：模板化生成

```
❌ 每次重新描述
"创建一个User类，包含id、name、email字段，
使用kotlinx.serialization，
所有字段可空除了id，
还要有created_at和updated_at..."
→ 每次都要重新描述，AI可能理解不一致

✅ 使用模板模式
"基于以下模式创建数据类：
1. 使用@Serializable注解
2. 参考User.kt的格式
3. 包含created_at和updated_at字段
现在创建CheckIn类"
→ AI学习模式后，后续生成速度提升3-5倍
```

**实际应用案例：**

```kotlin
// 第1次：详细描述
@Serializable
data class User(
    val id: String? = null,
    val auth_id: String? = null,
    val name: String,
    val emergency_email: String,
    val device_id: String? = null,
    val created_at: String? = null,
    val updated_at: String? = null
)

// 第2次：使用模式
@Serializable
data class CheckIn(
    val id: String? = null,
    val user_id: String,
    val check_in_date: String,
    val check_in_time: String? = null,
    val created_at: String? = null
)

// AI已经学会：
// - @Serializable注解
// - 可空字段的? = null语法
// - 时间戳字段的命名
```

### 优化3：增量迭代模式

```
❌ 大爆炸式开发（试图一次性完成所有功能）
第1轮：创建完整项目
→ 代码量大，错误多
→ 难以定位问题
→ 浪费时间

✅ 增量式迭代（先核心，后扩展）
第1轮：核心功能（用户创建 + 签到）
第2轮：状态持久化
第3轮：UI美化
第4轮：Edge Functions
第5轮：文档和测试

→ 每一步都可验证
→ 出错时容易定位
→ 可以随时调整方向
```

**实际应用案例：死了么项目的迭代过程**

```
第1轮：基础框架
✅ 创建项目结构
✅ 配置Gradle

第2轮：数据层

验证：可以插入和查询数据

验证：用户可以完成签到

第4轮：UI完善
✅ 添加加载状态
```

### 低效模式 vs 高效模式

```
场景：开发一个用户CRUD功能

❌ 低效模式：
第1轮：
用户：帮我写一个用户管理系统

用户：没有错误处理

AI：[再次修改数据模型]

→ 消耗token：3000+
→ 时间：30分钟
→ 成功率：60%
→ 心情：沮丧

✅ 高效模式：
第1轮：
用户：我要用Kotlin + Supabase实现用户CRUD
     先生成数据模型，使用kotlinx.serialization
AI：[生成User.kt，30行]
用户：✓ 正确

第2轮：
用户：继续生成Repository，包含CRUD操作
AI：[生成UserRepository.kt，80行]

第3轮：
用户：生成ViewModel，包含错误处理

```

```
   参考已有代码格式
   vs
   每次重新描述需求
   → 效率提升：1.5倍

总效率提升：2 × 1.5 × 2 × 1.5 = 9倍
```

## 🎓 实战：AI使用的黄金法则

### 法则1：明确输入，明确输出

```kotlin
// ❌ 模糊指令
"帮我处理用户签到"

问题：
- 用什么技术？
- 输入参数是什么？
- 返回值是什么？
- 如何处理错误？

// ✅ 明确指令
"创建一个函数 checkIn(userId: String): Result<CheckIn>
 需要实现：
 1. 调用 supabase.from('check_ins').insert()
 2. 传入 user_id 和当前日期
 3. 返回 Result<CheckIn> 类型
 4. 包含异常处理"

效果：
- AI知道确切要做什么
- 生成的代码符合预期
- 不需要反复修改
```

### 法则2：提供上下文，而非代码

```
❌ 错误做法：
"写一个签到界面，像微信那样"
→ AI需要猜测微信的界面是什么样的
→ 生成的结果可能不符合预期

✅ 正确做法：
"这是一个MVVM架构的Android应用，
 使用Jetpack Compose作为UI框架，
 数据层使用Supabase，
 现在创建签到界面，包含：
 1. 一个大的签到按钮
 2. 显示签到状态
 3. 显示签到统计信息"

效果：
- AI理解了整体架构
- 知道使用什么技术栈
- 明确了界面需求
- 生成的代码与项目风格一致
```

### 法则3：使用渐进式提示

```
第1轮：生成框架
提示："创建Android项目的基础结构"
输出：项目目录、配置文件、Gradle脚本
验证：项目可以编译 ✓

第2轮：实现核心功能
提示："实现用户创建和签到功能"
输出：数据模型、Repository、ViewModel
验证：功能可用 ✓

第3轮：完善细节
提示："添加错误处理和加载状态"
输出：UI状态管理、错误提示
验证：体验流畅 ✓

第4轮：优化体验
提示："添加动画和样式优化"
输出：过渡动画、颜色主题
验证：视觉美观 ✓

每一步都建立在前一步的基础上，风险可控
```

### 法则4：验证后再继续

```
✅ 正确的流程：
生成代码 → 人工review → 确认正确 → 继续下一步
           ↑______________|

第1步：生成User.kt
       ↓ review（字段正确、注解正确）
       ✓ 确认
第2步：生成UserRepository.kt
       ↓ review（逻辑正确、错误处理完善）
       ✓ 确认
第3步：生成UserViewModel.kt
       ↓ review（状态管理正确）
       ✓ 确认
→ 整个过程顺利，没有返工

❌ 错误的流程：
生成User.kt → User.kt → UserRepository.kt → UserViewModel.kt
            → UserScreen.kt → Navigation.kt
            ↓ 最后发现User.kt的字段定义错了
            → 需要修改所有依赖User.kt的文件
            → 陷入循环修改
```

### 法则5：使用工具而非依赖AI

```bash
# ❌ 完全依赖AI
用户：创建一个包含100个文件的目录结构
AI：[生成100个mkdir命令]

# ✅ 使用工具

或
用户：我可以使用tree命令查看结构吗？
```

## 💡 终极模式：AI作为"技术合伙人"

### 思维转变

```
❌ 传统思维：AI是"代码生成工具"
角色：工具
使用方式：你告诉它做什么，它执行
关系：单向命令
期望：它听我的话

✅ 现代思维：AI是"技术合伙人"
角色：合作伙伴
使用方式：你提出目标，它参与决策
关系：双向沟通
期望：共同解决问题
```

### 合伙人模式的特征

```
1. 双向沟通
   不是：
   用户：写代码
   AI：[写代码]

   而是：
   用户：我想实现签到功能
   AI：你想要iOS还是Android？
   用户：Android
   AI：推荐用Kotlin原生还是React Native？
   用户：Kotlin原生
   AI：好的，我们开始...

2. 主动建议
   不是：
   用户：这样写对吗？
   AI：对/不对

   而是：
   用户：我要实现用户认证
   AI：我建议使用Supabase的匿名认证，
      因为：
      1. 不需要用户注册
      2. 自动管理session
      3. 适合这个应用场景
      你觉得呢？

3. 知识共享
   不是：
   每次对话都重新介绍项目背景

   而是：
   AI了解项目的全局：
   - 技术栈：Kotlin + Supabase
   - 架构：MVVM + Repository
   - 风格：使用kotlinx.serialization
   - 命名：使用驼峰命名法
   → 后续生成的代码风格一致

4. 共同成长
   不是：
   AI每次都按默认模式生成

   而是：
   随着项目进行：
   - AI学习你的编码风格
   - AI记住你的偏好设置
   - AI理解你的业务逻辑
   → 后续效率越来越高
```

### 实际应用案例：死了么项目

```
我不仅仅是"执行命令"，而是作为"合伙人"：

   用户：按照教程进行初步复刻

      - 你有Supabase账号吗？
      - 想要iOS还是Android？
      - 要完整功能还是核心功能？
   → 确保方向正确

   AI：我为你创建了任务清单（11个步骤）
      - 创建Supabase项目
      - 配置数据库
      - 创建Android项目
      - ...
   → 提供全局视角

3. 质量控制
   AI：我建议添加RLS策略保护数据
      我建议使用匿名认证简化流程
      我建议创建视图简化查询
   → 主动提出改进建议

4. 文档完善
   AI：我创建了详细的SETUP_GUIDE.md
      包含：
      - 每个步骤的详细说明

   → 提供专业建议
```

### 工具 vs 合伙人的对比

```
场景：遇到技术问题

❌ 工具模式
用户：签到失败了，怎么办？
AI：检查网络连接
    检查API密钥
    检查权限设置
    [给出10个可能的解决方案]
→ 用户需要逐一尝试

✅ 合伙人模式
用户：签到失败了
AI：我看到你的代码，问题可能在于：
    1. Repository中没有处理唯一性约束冲突
    2. 前端没有正确显示错误信息

    我建议：
    - 在Repository中添加try-catch
    - 返回Result类型包裹错误
    - 在ViewModel中处理错误状态
    - 在UI中显示错误提示

    需要我帮你实现吗？
→ AI分析了代码，给出了针对性的解决方案
```

## 🎯 总结：AI使用的三层境界

```
第一层：工具使用者
特征：
- 把AI当代码生成器
- 给出简单指令
- 期望AI直接完成

使用方式：
"帮我写这段代码"
"修复这个bug"
"解释这个函数"

效果：
→ 效率提升：1.5倍
→ 适合：简单重复性任务
→ 局限：无法处理复杂项目

第二层：架构协作者
特征：
- 与AI共同规划
- 分步骤执行
- 及时验证反馈

使用方式：
"这是需求，帮我设计架构"
"我们一起实现这个功能"
"这样设计有什么问题？"

效果：
→ 效率提升：3-5倍
→ 适合：中等复杂度项目
→ 局限：需要人类主导所有决策

第三层：技术合伙人
特征：
- AI参与决策过程
- 主动提出建议
- 共同承担责任

使用方式：
"我们要构建这个产品，技术方案怎么选？"
"这个问题你怎么看？"
"我们一起解决这个挑战"

效果：
→ 效率提升：10倍+
→ 适合：任何复杂度的项目
→ 优势：释放人类创造力，聚焦业务价值
```

### 关键差异

```
工具使用者：
流程：
人类思考 → AI生成代码 → 人工修改 → AI再次修改
→ AI只是"手"，人类是"大脑"

架构协作者：
流程：
人类规划 → AI生成模块 → 人工组装 → AI优化
→ AI是"助手"，人类是"架构师"

技术合伙人：
流程：
共同决策 → 并行开发 → AI负责标准化 → 人类负责创新
→ AI是"合伙人"，人类是"产品负责人"
```

## 🎓 结语

"死了么"APP的成功复刻，不仅仅是一个技术项目的完成，更展示了**AI编程的正确打开方式**。

核心要点：

1. **技术选型比代码实现更重要**
   - 选择对AI友好的技术栈
   - 选择声明式而非命令式
   - 选择高自动化程度的服务

2. **使用模式比工具本身更重要**
   - 规划-执行-验证循环
   - 增量式迭代
   - 并行执行模式

3. **协作模式比单打独斗更有效**
   - AI作为技术合伙人
   - 人机分工明确
   - 共同成长

4. **思维模式转变是关键**
   - 从"AI是工具"到"AI是合伙人"
   - 从"命令AI"到"与AI协作"
   - 从"关注代码"到"关注价值"

在这个AI时代，**最大的杠杆不是AI本身，而是如何正确使用AI**。
